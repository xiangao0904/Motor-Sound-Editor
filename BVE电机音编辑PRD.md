## 项目背景  
BVE5的电机音制作软件

## 用户画像  
BVE车辆模组制作者、MTR模组车辆制作者

## 功能需求  (Functional Requirements)
本章节定义了 BVE5 Motor Assistance 软件的核心功能模块及交互规则。

### 主页 (Home Page / Project Management)
主页是软件的入口，主要负责工程文件的可视化管理、快速检索以及全局系统配置。

#### 工程卡片流 (Project Gallery)
+ **网格化展示**：以卡片形式排列显示所有历史工程（.bms 文件）。
+ **卡片信息详情**：
    - **预览图**：动态生成该工程内所有音轨曲线的缩略图（Pitch/Volume 叠加视图）。
    - **基本元数据**：显示文件名称（Name）、最后修改日期（Last Modified）。
+ **交互逻辑**：
    - **进入编辑**：左键点击卡片主体，立即跳转至该工程的“编辑页”。
    - **快捷操作**：右键点击显示隐藏菜单，支持“删除”或“导出”操作。

#### 工程检索与排序 (Search & Sorting)
+ **实时搜索框**：支持根据文件名进行模糊搜索，结果列表随输入实时过滤。
+ **多维排序**：提供“日期 (Date)”和“名称 (Name)”两种排序维度，默认按修改日期倒序排列。

#### 侧边导航栏 (Sidebar Navigation)
+ **新建工程 (New Project)**：点击后弹出对话框，输入工程名和最大速度（默认120）和加速度（默认1.2）后进入编辑页。
+ **打开文件 (Open File)**：调用系统文件浏览器，直接导入外部的 `.bms` 配置文件。
+ **系统设置 (Settings)**：
    - 暂时预留
+ **用户信息 (User Profile)**：右下角软件版本号（v0.0.1）。

###  编辑页 (Editor Page)
编辑页是模拟器的核心，集成了实时状态仿真、音轨配置与可视化曲线编辑。

####  状态模拟与实时显示 (Simulator Control)
+ **状态控制机**：
    - **牵引 (Traction)**：速度随 `Acceleration` 增加，至 `Max Speed` 停止。
    - **惰行 (Coasting)**：速度锁定当前值。
    - **制动 (Brake)**：速度按减速度下降至 0。
+ **速度控制与跳转**：
    - 左侧显示 `Current Speed` (km/h)。
    - **手动输入**：点击数值框输入新数值并回车，模拟器状态及画面指针同步跳转。
+ **播放游标 (Playhead)**：中央 Pitch/Volume 坐标系中各有一条**红色垂直游标**，其 X 轴位置与当前车速绝对绑定。

#### 音轨管理 (Track Management)
 音轨管理由“图层列表 (Track Layers)”和“属性详情 (Track Details)”两部分组成，位于编辑器的右侧侧边栏。  

##### 图层列表 (Track Layers Panel)
+ **列表化展示**：以纵向列表形式排列所有已导入的音轨，支持通过鼠标点击切换当前**激活音轨**，再次点击激活的音轨取消激活，进入 `None` 模式。
+ **图层视觉标识**：每个图层左侧显示其自定义的颜色块，方便用户在可视化编辑器中快速对应。
+ **快速开关控制器**：
    - **可见性开关 (Visibility)**：点击“眼睛”图标可显示/隐藏该音轨在可视化编辑器中的线。
    - **音频监听开关 (Mute/Solo)**：点击“小喇叭”图标控制该音轨在音频引擎中的静音状态。
+ **激活逻辑**：
    - 选中的音轨呈高亮状态，下方的“属性详情”面板同步刷新为该音轨的数据。
    - **快捷取消**：使用 `Ctrl+D` 清除当前选中的音轨，进入 `None` 模式。

##### 属性详情 (Track Details Panel)
该面板针对当前**选中的音轨**或**选中的关键帧**进行精确配置。

+ **音轨基本信息**：
    - **Name (名称)**：支持直接修改音轨在列表中的显示名称。
    - **Color (颜色)**：通过 Hex 色值输入框或拾取器修改曲线显示颜色（如 `FFE796`）。
    - **File (文件路径)**：显示当前关联的 `.wav` 或 `.ogg` 文件名，点击右侧“文件夹图标”可重映射音频资源。
+ **关键帧精确编辑**：
    - 当用户在可视化编辑器中选中某个关键帧（Anchor Point）时，此处会显示其精确坐标：
        * **X 轴**：对应车速 ($km/h$)。
        * **Y 轴**：对应参数值（Pitch 为倍率，Volume 为 0.0-1.0 增益）。
    - 用户可直接在输入框内修改数值，曲线将实时重绘。
+ **删除操作**：
    - **Delete Selected Point**：点击按钮删除当前在图中选中的单个关键帧。

####  可视编辑器 (Visual Editor)
编辑器分为 **Speed-Pitch** (上) 和 **Speed-Volume** (下) 两部分。针对 BVE 引擎牵引/制动关键帧分离的特性，编辑器显示逻辑与模拟器状态强绑定。

+ **状态联动显示逻辑**：
    - 当模拟器处于 **牵引 (Traction)** 模式时，图表仅渲染并允许编辑当前激活音轨的 **牵引关键帧**。
    - 当模拟器处于 **制动 (Brake)** 模式时，图表仅渲染并允许编辑当前激活音轨的 **制动关键帧**。
+ ** 插值逻辑  **
    - 两个相邻关键帧 $ (x_1, y_1) $ 与 $ (x_2, y_2) $ 之间仅由直线段连接。
    - 对于任意当前速度 x，其对应的参数 y 计算公式为：
    - $ y = y_1 + (x - x_1) \cdot \frac{y_2 - y_1}{x_2 - x_1} $
    -  编辑器折线无任何贝塞尔平滑处理，确保视觉呈现与音频引擎的线性步进完全一致。  

##### 左侧工具栏
+ **A. 选择工具 (Select Mode)**：
    - **悬浮高亮**：鼠标滑过未激活曲线时轻度高亮；点击即切换激活状态。
    - **锚点焦点**：激活态下，鼠标悬浮关键帧变红并放大。
    - **全局右键菜单**：在编辑区域内右键点击，弹出全局上下文菜单。包含核心选项 **“打开列表视图”**（用于精准修改和同步牵引/制动数据），其余选项暂时预留。  
    - **属性面板**：选中关键帧后，右侧显示精确的 (X, Y) 坐标供用户输入修改。
+ **B. 移动工具 (Move Mode)**：
    - **2D 拖拽**：支持左键按住关键帧在各自图表区域内自由移动（跨区域限制）。
    - **自动重排序 (Auto-Sorting)**：当关键帧 X 轴顺序发生交叉时，系统在释放鼠标的一瞬间按 X 轴重新排序并更新连线逻辑。
+ **C. 关键帧工具 (Keyframe Mode)**：
    - **左键添加**：在激活音轨曲线上任意点击，系统根据 X 轴插值生成新节点并刷新连线。
    - **右键删除**：点击激活态的现有关键帧即可删除，前后节点自动闭合。

####  列表视图编辑器 (List View Editor)  
为解决牵引与制动关键帧高度复用但分离存储的问题，提供基于表格的精准编辑与同步工具。

+ **入口**：在可视编辑器空白处右键 -> 选择“打开列表视图”。该视图基于当前激活的音轨弹出。
+ **界面布局**：提供对照式数据表格，将牵引与制动的 Speed、Volume、Pitch 数据并排展示，样式与软件整体风格匹配：

| traction |  |  |  | brake |  |  |  |
| --- | --- | --- | --- | --- | --- | --- | --- |
| speed | vol | speed | pit | speed | vol | speed | pit |
| 0 | 0 | 0 | 1 | 0 | 0 | 0 | 1 |
| 10 | 1 | 10 | 1.5 | 10 | 1 | 10 | 1.5 |
|  |  | 30 | 2 |  |  | 30 | 2 |
|  |  | 80 | 2.5 |  |  | 80 | 2.5 |
|  |  | 120 | 3 |  |  | 120 | 3 |


+ **交互与修改规则**：
    - **直接编辑**：用户可单击表格内的数值直接进行修改。若修改了 `Speed` 的数值，系统会立即对该列数据按速度升序**自动重新排序**。
+ **区域右键菜单 (Context Menus)**：
    - **牵引区 (Traction) 右键菜单**：
        1. **将牵引数据同步至制动**：一键将牵引侧所有关键帧数据无损覆盖到制动侧。
        2. **添加关键帧**：弹出次级操作，用户需勾选类型（Volume 关键帧 / Pitch 关键帧），输入速度与对应数值后插入，系统自动重排。
        3. **删除当前关键帧**：仅当鼠标悬浮在某个具体的列表项上方时可用，点击即删除该行；鼠标未悬浮于具体数据项时，该按钮呈灰色禁用状态。
    - **制动区 (Brake) 右键菜单**：逻辑同上，但选项 1 变为 **将制动数据同步至牵引**。
+ **底部操作按钮 (Actions)**：
    - **Cancel (取消)**：点击后直接关闭列表视图面板，丢弃在此面板内的所有修改，底部编辑页无任何变化。
    - **Apply (应用)**：点击后保存表格内的所有修改，关闭列表视图，并**立即同步刷新**底部的可视化曲线编辑页面。

### 导出与兼容性 (Export & Compatibility)

#### 3.4.1 .bms 专属工程格式定义
为了解决音频工程中常见的“素材丢失”和“路径断裂”问题，`.bms` 采用**容器化封装架构**。

+ **文件结构：**`.bms` 本质上是一个经过压缩的二进制容器，内部包含以下部分：
    - **project.json (元数据)：** 记录软件版本号、作者信息、最近修改日期、最大速度限制等全局参数。
    - **tracks.json (曲线数据)：** 核心逻辑文件。以 JSON 格式存储所有音轨的关键帧坐标 (Speed, Value)、插值算法类型（线性/平滑）、音轨颜色设定及激活状态。
    - **Assets/ (资源文件夹)：** 存放该工程引用的所有原始音频文件（.wav 或 .ogg）。
+ **封装逻辑：** 当用户导入外部音频时，软件会将其**副本**拷贝至内存缓冲区；在执行 Ctrl+S 保存时，这些音频将被统一打包进 `.bms` 文件内，而非仅记录本地路径。ctrl+shift+s实现另存为

#### 3.4.2 工程存储与共享流程
+ **保存机制 (Save/Save As)：**
    - 系统执行增量或全量压缩，将最新的关键帧连线 JSON 与资源文件夹打包。
    - `.bms` 文件作为单一文件输出，便于文件系统管理。
+ **共享与便携性 (Sharing & Portability)：**
    - **无损传递：** 用户 A 将 `.bms` 发送给用户 B 后，用户 B 只需双击打开，软件会自动解压临时资源到缓存区（或者使用openfile功能）。
    - **环境无关性：** 由于所有音频素材已内置，用户 B 看到的编辑器状态（音轨、曲线、实时预览效果）将与用户 A 完全一致，无需重新定位素材路径。
+ **读取逻辑 (Open Project)：**
    - 校验文件完整性与版本兼容性。
    - 若 `.bms` 内部版本高于当前软件版本，提示用户升级软件；若低于当前版本，执行自动迁移/转换逻辑。

#### 3.4.3 预览与编辑同步
+ **即时预览：** 工程打开后，音频引擎立即加载 `Assets/` 中的素材，用户点击播放游标即可听到完整的电机音合成效果。

## 非功能需求 (Non-Functional Requirements)
本章节定义了系统在性能、平台兼容性、稳定性以及可用性方面的质量标准。

### 4.1 性能需求 (Performance)
作为一款涉及实时音频生成与动态图表绘制的软件，性能是重中之重。

+ **UI 渲染流畅度**：
    - 主视口的游标（红线）在列车加减速模拟期间，必须保持至少 **60 FPS** 的平滑移动，不可出现视觉卡顿或跳帧。
    - 关键帧的拖拽、曲线的重绘延迟需控制在 **16ms** 以内（即视觉无延迟）。
+ **音频处理低延迟 (Low Latency)**：
    - 当用户在播放状态下拖拽关键帧，或输入新数值时，音频引擎的变调/增益反馈延迟需低于 **50ms**。
    - 多音轨（10+ 条）并发播放并实时进行重采样（Resampling）计算时，不应出现爆音（Clicking/Popping）或音频缓冲区欠载（Buffer Underrun）。

### 4.2 可靠性与容错 (Reliability & Fault Tolerance)
+ **状态撤销栈 (Undo/Redo)**：支持至少 50 步的历史操作撤销功能（快捷键 `Ctrl+Z` / `Ctrl+Y`），这对于参数调试工具是绝对的刚需，然后。
+ **崩溃恢复 (Crash Recovery)**：系统需具备定时自动保存功能（例如每 3 分钟），在遭遇意外崩溃后，下次启动能提示用户恢复未保存的工作区。
+ **容错性**：当导入不支持的音频格式、损坏的文件，或输入非法的坐标数值（如字母代替数字）时，系统需进行异常拦截，并给予友好的 UI 弹窗提示，绝不能直接闪退。

### 4.3 平台与部署需求 (Platform & Deployment)
+ **主力操作系统**：完全支持 Windows 10 / Windows 11 (64-bit)。
+ **多端兼容（加分项）**：架构设计上应预留跨平台（macOS / Linux）的编译可能，但不作为首发版本的强制阻碍。
+ **免安装/便携性**：考虑到 BVE5 玩家群体的习惯，最好能提供绿色免安装版（Portable Zip），解压即用，不强依赖复杂的系统运行库（或将依赖打包在内）。

### 4.4 易用性需求 (Usability)
+ **全局热键系统**：高度依赖快捷键（如 `Space` 播放/暂停，`Ctrl+D` 取消焦点，`Delete` 删除选中节点，`Ctrl+S` 保存等），提升高频调试效率。
+ **深色模式优先**：考虑到开发者/音频处理者通常在暗光环境下长时间工作，强制全局深色主题（已在 UI 设计中体现），以减轻视觉疲劳。

## 技术架构选型 (Technical Architecture)
为满足 UI 高度定制化、跨平台潜能以及较低的系统资源占用，本项目采用 **Tauri (Web 前端 + Rust 后端)** 架构进行开发。

### 5.1 整体架构视图
系统采用前后端分离的桌面端应用架构（B/S 变体）：

+ **UI 层 (Frontend)**：负责所有界面渲染、用户交互、图表绘制以及状态管理。
+ **系统层 (Backend / Tauri Core)**：负责操作系统原生的文件系统访问、窗口管理以及可能的高性能音频流处理。
+ **通信桥 (IPC)**：前端通过 Tauri 提供的 IPC (Inter-Process Communication) 机制与底层 Rust 侧进行数据交换（如读取大容量音频文件）。

### 5.2 前端技术栈 (Frontend Stack)
前端全权负责 UI 交互、数据可视化与初期音频引擎，采用现代化的 Web 技术栈构建：

+ **核心框架与构建工具**：采用 **Vue 3 (Composition API)** 配合 **Vite** 构建。
    - **Vite**：提供极速的冷启动与模块热替换 (HMR)，在频繁调试 UI 细节和交互逻辑时确保极佳的开发体验。
    - **TypeScript**：实施严格的静态类型检查。考虑到 BVE5 `sound.cfg` 数据结构的解析、关键帧二维坐标 `(x, y)` 的实时运算，TS 能显著降低数据流转过程中的类型错误，提升代码可维护性。
+ **状态管理 (State Management)**：采用 **Pinia**。负责全局核心业务数据的响应式管理，主要包括：
    - **模拟器状态**：当前列车速度、运行模式（牵引/惰行/制动）。
    - **音轨状态**：数十条音轨的配置信息、激活/焦点状态、Mute/Solo 状态。
    - **历史记录栈**：维护用户操作步骤，实现撤销/重做 (Undo/Redo) 功能。
+ **图表与可视化引擎 (Visual Editor)**：
    - **主视口渲染**：采用 轻量级的 2D 渲染引擎（ Konva.js）。 由于全是直线段，绘制逻辑简化为 `Line` 组件，通过 `points` 数组直接渲染。  
    - **CSS扩展：**使用Tailwind CSS 
+ **音频引擎 (Audio Engine - Phase 1)**：
    - 采用浏览器原生的 **Web Audio API**。
    - 利用 `AudioBufferSourceNode` 实现音频采样读取与实时变调（基于 `playbackRate`）。
    - 利用 `GainNode` 实现多音轨音量的实时平滑调制。
    - 结合 Vue/Pinia 的状态下发，将当前模拟速度 (X轴) 实时映射为音频节点的 Y轴控制参数。
    -  插值公式计算出瞬时的 `playbackRate` (Pitch) 和 `Gain` (Volume)。  

### 5.3 后端技术栈 (Backend Stack)
+ **核心框架**：Tauri (Rust)。
+ **打包与部署**：依赖 Tauri 自带的打包工具，直接编译为 Windows `.exe` 单文件或 `.msi` 安装包。
+ **文件系统交互**：利用 Tauri 的 `fs` 模块，绕过浏览器的沙盒限制，实现对用户本地目录中 BVE5 工程文件 (`sound.cfg`) 和 `.wav` 文件的无缝读取与写入。

### 5.4 架构演进路线 (Architecture Roadmap)
+ **初期阶段**：音频处理逻辑强依赖前端 Web Audio API，Rust 端仅负责文件 I/O 与窗口壳子。实现最快的产品 MVP (最小可行性产品)。
+ **后期优化阶段 (预留)**：若发现 JS 线程在处理多达 20-30 条高采样率音轨时出现性能瓶颈，再考虑引入 Rust 的 `rodio` 等音频库，将音频重采样和混音下沉至底层处理，前端仅发送控制指令。
